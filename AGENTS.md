# Codex Project Guide

本项目是一个本地运行的北京家庭财务规划工具，覆盖收入、支出、购房、买车、理财、贷款、账户曲线和事件时间线。前端使用 React + Vite，后端使用 FastAPI + SQLite。代码库应保持为可发布、可复用的通用规划工具；个人家庭配置只应保存在本机数据库或本机私有脚本中。

## 目录结构

- `backend/app/schemas.py`：Pydantic 数据模型、API 请求和响应结构。
- `backend/app/calculator.py`：后端计算编排入口，负责组织输入、调用政策包和领域算法、汇总账户推演、策略、事件和可视化序列。新增纯算法逻辑应优先下沉到 `backend/app/domain/`。
- `backend/app/domain/`：基础领域算法模块，包括贷款、时间、车辆、养娃、阶段性支出、税务基础、职业冲击、投资税务和幸福指数等可复用计算。
- `backend/app/policies.py`：政策抽象和北京政策包。新增地区政策时优先扩展这里的接口。
- `backend/app/database.py`：SQLite 存取、初始化和迁移。默认数据库位于 `%APPDATA%\house-planner\planner.db`，可用 `HOUSE_PLANNER_DB` 覆盖。
- `backend/app/main.py`：FastAPI 路由。
- `backend/tests/`：后端 API、计算器和政策逻辑测试。
- `frontend/src/App.tsx`：主要前端页面和交互逻辑。
- `frontend/src/types.ts`：前端类型，应与 `backend/app/schemas.py` 对齐。
- `frontend/src/api.ts`：前端 API 调用。
- `frontend/src/styles.css`：全局样式、视觉语言、浅色/深色主题。
- `scripts/privacy_scan.py`：提交或推送前的隐私扫描。
- `scripts/push_public.ps1`：发布检查和推送脚本。
- `docs/architecture.md`：开发者架构说明。较大重构、跨模块修改、接手陌生逻辑前应先阅读这里，理解前后端边界、核心数据模型、计算生命周期、数据库迁移、政策包和策略生成关系。

## 核心设计原则

- 后端是计算真源。现金余额、账户余额、投资收益、税、月供、贷款余额、策略生成、事件时间线、可视化序列都应由后端生成；前端只负责展示、编辑输入参数和发起重算。
- 前端不要保留旧版兜底计算。类似 `calculatedMonthlySeries` 这类旧本地推演逻辑不应继续承担业务计算。
- 策略应同时支持自动推荐和手动调整。自动策略要解释依据、政策约束、现金安全、关键取舍；手动策略要进入同一套后端推演，不能只在前端临时显示。
- 账户是基本概念。至少区分现金账户、投资账户、公积金账户、固定资产、贷款账户。公积金账户因政策限制不应混同为可自由现金。
- 贷款是基本概念。房贷、商贷、公积金贷、车贷和其他当前贷款都应进入统一贷款模型，并能输出余额、月供、利息、提前还款或冲还贷事件。
- 政策规则应通过接口抽象。北京政策是一个政策包，不要把地区政策常量散落在 UI 和业务流程里；政策上下限、年龄、期限、房龄、税费、专项扣除和公积金规则应可替换。
- 可视化展示的是当前选中方案的完整推演，不是当前静态状态。房源、车辆、理财策略、职业冲击和手动参数变化后，应触发后端重算并显示计算中状态。
- 买房和买车属于同一类“重大消费目标”。交互模型应尽量统一为：目标资产、购买时机、资金策略、贷款策略、后续持有成本、是否纳入当前规划。房源和车辆可以保留各自专业参数，但增删、复制、停用、排序、策略生成和影响预览不应各搞一套。
- 多目标保存必须以目标列表为单位。前端保存房源目标时不能只保存当前选中的一个；车辆需求同理应写回完整 `car_plan.vehicle_plans`。删除或停用目标时，前端应先更新本地列表并显示重新计算状态，再等待后端保存和重算结果。
- 跨模块重构时同步维护 `docs/architecture.md`。如果修改了核心数据模型、计算请求生命周期、数据库迁移方式、政策包接口、重大消费目标模型或前后端分工，应在同一轮更新架构文档，避免 README、AGENTS 和实际代码分叉。

## 隐私与发布原则

- 不要提交本地 SQLite 数据库、导出文件、截图、日志、`.env`、虚拟环境、依赖目录或构建产物。
- 不要在代码、测试、README、AGENTS、提交信息里写入真实家庭成员称呼、收入、资产、债务、出生年月、单位、学校、身份或其他私人信息。
- 测试样例只使用泛化名称，例如 `样例成员A`、`当前贷款A`、`示例房源`、`示例车辆`。
- 内部字段使用通用业务概念，不使用会暗示个人真实债务来源的命名。比如统一使用 `current_loans`、`phased_loans`、`loan_category` 等通用概念。
- 旧字段如果带有隐私暗示，不要长期保留兼容读取逻辑；需要迁移时优先做一次性本地数据转换，并确保转换脚本不进入公开仓库。
- 北京政策、公开规则和官方链接可以保留；个人配置只留在本机数据中。
- 不要频繁推送。除非用户明确要求，完成工作后只做本地修改和本地验证。
- 发布前必须跑隐私扫描；推送优先使用 `.\scripts\push_public.ps1`，不要绕过 pre-push 保护。

## 编码与中文乱码防护

- 代码、文档、测试和脚本统一按 UTF-8 保存。发现中文显示异常时，先判断是终端显示问题还是文件内容真的损坏，不要把终端里的乱码输出复制回源码。
- Windows PowerShell 通过管道把含中文脚本传给 Python/Node 时，容易因为控制台编码、子进程 stdin/stdout 编码和文件编码不一致，把正常中文临时显示成乱码，甚至把脚本源码里的中文样本污染。含中文脚本优先保存为 UTF-8 文件后执行；临时管道脚本尽量保持 ASCII，必要时使用 Unicode 转义。
- 过去反复出现的编码事故根因主要有四类：把终端显示乱码误判为源码乱码；把含中文或复杂引号的 Python/Node 脚本直接塞进 PowerShell 管道；在 `apply_patch` 里用已经乱码或含替换符的中文行做精确锚点；从失败的终端输出里复制文本回源码。这些都必须按下面流程规避。
- 不得仅凭命令输出里的 UTF-8/GBK 误读片段、`U+FFFD` 替换符或问号断言“源码中文常量已经损坏”。先用至少一种文件级验证确认：`python scripts/encoding_scan.py`、`Get-Content -Encoding utf8 <文件>`、或用 Python `Path(...).read_text(encoding="utf-8")` 读取目标文件。如果扫描通过而终端显示异常，结论应写成“终端/命令输出编码异常”，不要说源码已损坏。
- 需要脚本处理中文、正则替换中文、或生成含中文内容时，不要使用 PowerShell here-string、管道 stdin、复杂嵌套引号或一长串内联 `python -c`。优先用 `apply_patch` 做源码修改；确需脚本时，把脚本保存为明确 UTF-8 编码的临时文件后执行，或让脚本主体保持 ASCII 并通过文件路径/JSON/Unicode 转义传参。
- `apply_patch` 匹配失败时，不要反复拿乱码中文或含替换符的行做锚点。改用稳定的 ASCII 标识符、函数名、类型名、字段名定位，扩大到整个函数/配置块替换；补丁失败后先重新读取文件确认最新上下文，再做下一次补丁。
- PowerShell 命令里如果涉及中文路径、中文文本、正则替换或多层引号，避免把读取、转换、写入、测试串成一条复杂命令。拆成可验证的小步骤；每一步只做一种事，先确认输出再继续。
- 修复中文断言或中文文案时，不要从终端乱码输出复制期望字符串。应从源码 UTF-8 读取结果、产品界面实际文案或明确的新文案重新写入；测试锚点优先使用函数名、英文标识符和结构化字段。
- 联网、测试、扫描或生成文件前，如命令涉及中文路径、中文字符串或 Python/Node 管道，先设置：

```powershell
$OutputEncoding = [System.Text.UTF8Encoding]::new($false)
[Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$Env:PYTHONIOENCODING = "utf-8"
```

- 检查中文乱码时运行 `python scripts/encoding_scan.py`。该脚本会严格验证文本文件是否可用 UTF-8 解码，并扫描替换符、问号替代串和 UTF-8/GBK 互相误读后的异常片段。
- `scripts/encoding_scan.py` 自身不要写入原始乱码样本，使用 Unicode 转义维护样本，避免扫描脚本误报自己。
- 如果扫描结果为 0 但终端仍显示乱码，优先修命令编码或输出方式；如果扫描命中真实源码行，再用 `Get-Content -Encoding utf8`、Python UTF-8 读取或编辑器确认上下文后修复。
- 每次修改含中文的源码、测试或文档后，必须运行 `python scripts/encoding_scan.py`，并用 `git diff -- <文件>` 快速检查没有引入 `U+FFFD` 替换符、问号替代串或 UTF-8/GBK 互相误读后的片段。

## 后端规则

- 新增或修改持久化字段时，同步更新 schema、当前数据库格式、默认初始化、API 测试和前端类型。
- 数据库格式变化时，应把运行时数据库结构和 JSON 记录一次性转换到最新格式，并清理旧字段、旧缓存和旧迁移记录；不要长期叠加历史迁移链或保留旧格式兼容读取逻辑。
- 本地数据转换必须可重复执行、先备份、保留用户已有配置。不能用直接删除或重建真实数据库代替迁移，除非用户明确要求。
- 数字字段要在 schema 中设置合理校验，例如 `min`、`max`、`step` 对应的业务范围；前端校验只是体验层，后端校验必须存在。
- 自动策略生成不应先生成明显不可行方案再让用户“修现金安全”。生成阶段就应尽量避免现金缺口；无法避免时要标出缺口、原因和调整路径。
- 年终奖按发放月一次性入账，不均摊到每个月。发放月应支持按成员、按收入阶段配置。
- 工资阶段用于模拟换工作、收入变化、年终奖变化、公积金比例变化、社保缴纳方式变化和生效时间。
- 收入阶段类型要驱动默认配置：工资就业默认开启工资社保扣缴并关闭自由职业收入；自由职业默认关闭工资社保扣缴并开启自由职业收入；失业期和养老金阶段默认关闭两者；手动阶段允许用户自行覆盖。阶段类型联动应一次性 patch 更新，避免连续状态更新互相覆盖。
- 当前贷款应是通用模型，能覆盖房贷、车贷、消费贷、教育类贷款等不同来源；还款方式、只还息阶段、利率、期数和生效时间都应可配置。
- 投资策略要考虑现金安全垫、动态定投、买入卖出手续费、收益复投、必要时卖出补充现金，以及买房买车等大额事件。
- 现金账户不能被解释为可为负余额。推演低于 0 时应显示为现金缺口，并让方案进入不可行或需调整状态。
- 车贷“前 X 期免息”按厂家或经销商贴息建模，不按合同免息建模。贷款余额应按合同全期等额本息推演，家庭现金流记录贴息后净还款；输出需区分合同月供、贴息额、家庭承担利息和提前还本节省利息。

## 政策规则

- 北京住房公积金贷款要考虑额度上限、缴存年限、首套/二套、房屋性质、新房/二手房、房龄、贷款年限、申请人年龄和政策规定的上下限。
- 公积金贷款利率和契税属于政策包职责。计算不得直接使用房源目标里的 `provident_rate` 或 `deed_tax_rate` 作为真源；应由 `policies.py` 按城市、首套/二套、贷款期限、面积等规则返回。商贷利率和中介费可作为市场报价或市场费用假设保留手动覆盖。
- 公积金账户变化要考虑缴存、利息、租房约定提取、购房相关提取、冲还贷和账户保留余额。
- 北京公积金冲还贷应按政策建模，不要用“每月随便提取到现金流”的方式处理。冲还贷支出应进入公积金账户曲线，并同步影响贷款余额。
- 税费、专项扣除、养老相关扣除、失业金和自缴社保等政策相关项，优先由政策包自动计算；只有个人选择或无法由政策判断的部分才交给用户手动配置。
- 政策描述要写清楚“依据什么规则”“本方案怎么应用”“哪些参数来自用户配置”“哪些参数来自政策包”。

## 前端与可视化规则

- 页面应按真实工作流组织，不要把不相关配置堆在同一页。家庭画像、收入支出、当前贷款、购房、买车、理财、政策规则和可视化应各自聚焦。
- 计划型页面统一使用同一骨架：顶部摘要区 -> 目标/方案列表 -> 当前选中项配置 -> 当前策略说明 -> 影响预览。购房、购车、理财、养娃、税务和导出都应遵循这个顺序，不要一页左右栏、一页上下排、一页纯表单。
- 目标或方案列表统一采用横向卡片。卡片负责选择、比较、启用、删除或采用；详情区只编辑当前选中项，不要把所有目标的完整表单纵向堆在页面里。
- 购房计划和买车计划页面都应先展示目标列表，而不是默认塞一个完整目标表单。默认没有买房或买车目标；用户通过“添加目标房源”或“添加车辆需求”开始。目标卡片应支持编辑、复制、删除、停用/启用，并显示规划顺序、状态和关键摘要。
- 购买顺序使用用户能理解的概念：自动排队、可并行考虑、手动指定时间、排在某目标之后等待若干月、暂不纳入规划。不要只用“第一套/第二套”“第一辆/第二辆”这类固定命名驱动逻辑。
- 输入框要兼容中文输入法组合输入。不要在普通文本输入的每次按键中做破坏性的格式化或强制转换。
- 前端显示文字使用中文；不要把后端英文字段名暴露给用户。图表 tooltip、空状态、导出文本和错误信息都要检查。
- API 和网络异常应转成用户能理解的中文提示，不直接显示 `Failed to fetch`、后端字段名或英文堆栈。
- 图标按钮必须有可访问名称，例如 `aria-label` 或清晰的可见文字；纯装饰图标不要承担隐藏语义。
- 开关和输入的布局要按意义分组且对齐。类似“工资社保扣缴 / 自由职业收入”这类同一类别的开关，桌面端应在同一行呈现；移动端再折为单列。阶段类型变化带来的自动配置应清楚但不剥夺用户手动覆盖。
- 可视化页应按“决策故事线”组织，而不是纯图表堆叠：结论摘要、关键变化归因、账户曲线、月现金流、贷款、公积金、事件时间线、方案对比。
- 联动月份查看应贯穿可视化页面。月份选择要有刻度、可直接选择年月，并尽量在滚动时保持可见；曲线窗口应支持图形化调整位置和宽度。
- 图表需要明确图例、颜色区分和说明。深色模式下要检查背景、坐标轴、网格线、tooltip、表格、按钮、输入框、选中态和警告色的对比度。
- 月现金流要区分生活支出、家庭支持、贷款、车辆、保险、保养、税费、投资买入卖出等项目；年度或季度支出不要错误均摊，除非解释为预算口径。
- 资产展示按账户逻辑命名：现金账户、投资账户、固定资产。公积金账户单独展示，贷款余额和月供单独展示。
- 方案对比应像决策表，突出买入时间、现金缺口、贷款结构、总利息、现金安全、幸福指数、关键风险和推荐理由。

## 视觉与交互风格

- 全局视觉语言要统一：同类按钮、卡片、表单、图表、标签和状态提示使用一致的间距、圆角、边框、阴影和颜色语义。
- 颜色语义固定：现金/流动资产为绿色，贷款/压力/支出为红橙，政策/规则为蓝色，投资为紫或青紫，固定资产为金色，警告为橙色，成功为绿色。浅色和深色模式只调整明度和对比度，不改变语义。
- 控件尺寸固定使用全局变量；输入框、下拉框、开关、按钮在同一行时基线必须对齐。桌面端一行承载 2-4 个相关字段，移动端再折叠成单列。
- 页面第一屏应是实际工具，不做营销式落地页。
- 不要使用单一色相铺满全站。图表颜色要有足够区分度，浅色和深色主题都要可读。
- 页面排版优先清晰、密度适中、适合反复查看。避免大片空白、无意义指标、嵌套卡片和说明文字压过核心决策信息。
- 文字解释要像财务顾问的说明，而不是字段定义。重点回答“为什么这样算”“这对家庭意味着什么”“改变哪个参数会影响结果”。
- 前端布局变更至少用正常桌面浏览器比例和移动端比例各检查一次。桌面端以常见笔记本宽度为准，移动端检查顶部导航、卡片列表、按钮高度、输入框对齐、错误提示换行和图表容器不溢出。

## 完成工作前检查

- 后端逻辑、schema、数据库或政策包变化：运行后端 pytest。
- 前端类型、页面、交互或可视化变化：运行 `npm run build`，必要时用浏览器检查关键页面。
- 中文文案、脚本、README 或 AGENTS 变化：运行 `python scripts/encoding_scan.py`。
- 架构、核心模型、数据库、策略生成或前后端职责变化：同步检查并更新 `docs/architecture.md`。
- 涉及 Git 的任何改动：运行 `python scripts/privacy_scan.py`。
- 准备发布：运行 `.\scripts\push_public.ps1`，不要手动绕过隐私扫描和推送保护。

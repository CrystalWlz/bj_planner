# Codex Project Guide

本项目是一个本地运行的北京家庭财务规划工具，覆盖收入、支出、购房、买车、理财、贷款、账户曲线和事件时间线。前端使用 React + Vite，后端使用 FastAPI + SQLite。代码库应保持为可发布、可复用的通用规划工具；个人家庭配置只应保存在本机数据库或本机私有脚本中。

## 目录结构

- `backend/app/schemas.py`：Pydantic 数据模型、API 请求和响应结构。
- `backend/app/calculator.py`：核心测算逻辑，包括账户推演、税费、贷款、策略生成、事件和可视化序列。
- `backend/app/policies.py`：政策抽象和北京政策包。新增地区政策时优先扩展这里的接口。
- `backend/app/database.py`：SQLite 存取、初始化和迁移。默认数据库位于 `%APPDATA%\house-planner\planner.db`，可用 `HOUSE_PLANNER_DB` 覆盖。
- `backend/app/main.py`：FastAPI 路由。
- `backend/tests/`：后端 API、计算器和政策逻辑测试。
- `frontend/src/App.tsx`：主要前端页面和交互逻辑。
- `frontend/src/types.ts`：前端类型，应与 `backend/app/schemas.py` 对齐。
- `frontend/src/api.ts`：前端 API 调用。
- `frontend/src/styles.css`：全局样式、视觉语言、浅色/深色主题。
- `scripts/privacy_scan.py`：提交或推送前的隐私扫描。
- `scripts/push_public.ps1`：发布检查和推送脚本。

## 核心设计原则

- 后端是计算真源。现金余额、账户余额、投资收益、税、月供、贷款余额、策略生成、事件时间线、可视化序列都应由后端生成；前端只负责展示、编辑输入参数和发起重算。
- 前端不要保留旧版兜底计算。类似 `calculatedMonthlySeries` 这类旧本地推演逻辑不应继续承担业务计算。
- 策略应同时支持自动推荐和手动调整。自动策略要解释依据、政策约束、现金安全、关键取舍；手动策略要进入同一套后端推演，不能只在前端临时显示。
- 账户是基本概念。至少区分现金账户、投资账户、公积金账户、固定资产、贷款账户。公积金账户因政策限制不应混同为可自由现金。
- 贷款是基本概念。房贷、商贷、公积金贷、车贷和其他当前贷款都应进入统一贷款模型，并能输出余额、月供、利息、提前还款或冲还贷事件。
- 政策规则应通过接口抽象。北京政策是一个政策包，不要把地区政策常量散落在 UI 和业务流程里；政策上下限、年龄、期限、房龄、税费、专项扣除和公积金规则应可替换。
- 可视化展示的是当前选中方案的完整推演，不是当前静态状态。房源、车辆、理财策略、职业冲击和手动参数变化后，应触发后端重算并显示计算中状态。

## 隐私与发布原则

- 不要提交本地 SQLite 数据库、导出文件、截图、日志、`.env`、虚拟环境、依赖目录或构建产物。
- 不要在代码、测试、README、AGENTS、提交信息里写入真实家庭成员称呼、收入、资产、债务、出生年月、单位、学校、身份或其他私人信息。
- 测试样例只使用泛化名称，例如 `样例成员A`、`当前贷款A`、`示例房源`、`示例车辆`。
- 内部字段使用通用业务概念，不使用会暗示个人真实债务来源的命名。比如统一使用 `current_loans`、`phased_loans`、`loan_category` 等通用概念。
- 旧字段如果带有隐私暗示，不要长期保留兼容读取逻辑；需要迁移时优先做一次性本地数据转换，并确保转换脚本不进入公开仓库。
- 北京政策、公开规则和官方链接可以保留；个人配置只留在本机数据中。
- 不要频繁推送。除非用户明确要求，完成工作后只做本地修改和本地验证。
- 发布前必须跑隐私扫描；推送优先使用 `.\scripts\push_public.ps1`，不要绕过 pre-push 保护。

## 后端规则

- 新增或修改持久化字段时，同步更新 schema、数据库迁移、默认初始化、API 测试和前端类型。
- 数据库迁移要可重复执行，并保留用户已有本地配置。不能用重建数据库代替迁移，除非用户明确要求。
- 数字字段要在 schema 中设置合理校验，例如 `min`、`max`、`step` 对应的业务范围；前端校验只是体验层，后端校验必须存在。
- 自动策略生成不应先生成明显不可行方案再让用户“修现金安全”。生成阶段就应尽量避免现金缺口；无法避免时要标出缺口、原因和调整路径。
- 年终奖按发放月一次性入账，不均摊到每个月。发放月应支持按成员、按收入阶段配置。
- 工资阶段用于模拟换工作、收入变化、年终奖变化、公积金比例变化、社保缴纳方式变化和生效时间。
- 当前贷款应是通用模型，能覆盖房贷、车贷、消费贷、教育类贷款等不同来源；还款方式、只还息阶段、利率、期数和生效时间都应可配置。
- 投资策略要考虑现金安全垫、动态定投、买入卖出手续费、收益复投、必要时卖出补充现金，以及买房买车等大额事件。
- 现金账户不能被解释为可为负余额。推演低于 0 时应显示为现金缺口，并让方案进入不可行或需调整状态。

## 政策规则

- 北京住房公积金贷款要考虑额度上限、缴存年限、首套/二套、房屋性质、新房/二手房、房龄、贷款年限、申请人年龄和政策规定的上下限。
- 公积金账户变化要考虑缴存、利息、租房约定提取、购房相关提取、冲还贷和账户保留余额。
- 北京公积金冲还贷应按政策建模，不要用“每月随便提取到现金流”的方式处理。冲还贷支出应进入公积金账户曲线，并同步影响贷款余额。
- 税费、专项扣除、养老相关扣除、失业金和自缴社保等政策相关项，优先由政策包自动计算；只有个人选择或无法由政策判断的部分才交给用户手动配置。
- 政策描述要写清楚“依据什么规则”“本方案怎么应用”“哪些参数来自用户配置”“哪些参数来自政策包”。

## 前端与可视化规则

- 页面应按真实工作流组织，不要把不相关配置堆在同一页。家庭画像、收入支出、当前贷款、购房、买车、理财、政策规则和可视化应各自聚焦。
- 输入框要兼容中文输入法组合输入。不要在普通文本输入的每次按键中做破坏性的格式化或强制转换。
- 前端显示文字使用中文；不要把后端英文字段名暴露给用户。图表 tooltip、空状态、导出文本和错误信息都要检查。
- 可视化页应按“决策故事线”组织，而不是纯图表堆叠：结论摘要、关键变化归因、账户曲线、月现金流、贷款、公积金、事件时间线、方案对比。
- 联动月份查看应贯穿可视化页面。月份选择要有刻度、可直接选择年月，并尽量在滚动时保持可见；曲线窗口应支持图形化调整位置和宽度。
- 图表需要明确图例、颜色区分和说明。深色模式下要检查背景、坐标轴、网格线、tooltip、表格、按钮、输入框、选中态和警告色的对比度。
- 月现金流要区分生活支出、家庭支持、贷款、车辆、保险、保养、税费、投资买入卖出等项目；年度或季度支出不要错误均摊，除非解释为预算口径。
- 资产展示按账户逻辑命名：现金账户、投资账户、固定资产。公积金账户单独展示，贷款余额和月供单独展示。
- 方案对比应像决策表，突出买入时间、现金缺口、贷款结构、总利息、现金安全、幸福指数、关键风险和推荐理由。

## 视觉与交互风格

- 全局视觉语言要统一：同类按钮、卡片、表单、图表、标签和状态提示使用一致的间距、圆角、边框、阴影和颜色语义。
- 页面第一屏应是实际工具，不做营销式落地页。
- 不要使用单一色相铺满全站。图表颜色要有足够区分度，浅色和深色主题都要可读。
- 页面排版优先清晰、密度适中、适合反复查看。避免大片空白、无意义指标、嵌套卡片和说明文字压过核心决策信息。
- 文字解释要像财务顾问的说明，而不是字段定义。重点回答“为什么这样算”“这对家庭意味着什么”“改变哪个参数会影响结果”。

## 开发验证命令

后端计算和 API 验证：

```powershell
$Env:PYTHONPATH = "backend"
python -m pytest backend/tests/test_api.py backend/tests/test_calculator.py -q -n auto
```

前端构建：

```powershell
Push-Location frontend
npm run build
Pop-Location
```

隐私扫描：

```powershell
python scripts/privacy_scan.py --ref HEAD
python scripts/privacy_scan.py
```

发布脚本：

```powershell
.\scripts\push_public.ps1
```

本地公开预览可使用临时数据库，避免碰到真实本地数据：

```powershell
$Env:HOUSE_PLANNER_DB = "$Env:TEMP\bj_planner_public_preview.sqlite"
$Env:PYTHONPATH = "backend"
python -m uvicorn app.main:app --host 127.0.0.1 --port 8010
```

```powershell
Push-Location frontend
$Env:VITE_API_BASE = "http://127.0.0.1:8010"
npm run dev -- --host 127.0.0.1 --port 5175
Pop-Location
```

## 完成工作前检查

- 后端逻辑、schema、数据库或政策包变化：运行后端 pytest。
- 前端类型、页面、交互或可视化变化：运行 `npm run build`，必要时用浏览器检查关键页面。
- 涉及 Git 的任何改动：运行 `python scripts/privacy_scan.py`。
- 准备发布：运行 `.\scripts\push_public.ps1`，不要手动绕过隐私扫描和推送保护。
